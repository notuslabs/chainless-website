## MDX Strategy: next-mdx-remote/rsc

**Decision: Use `next-mdx-remote` (RSC variant) over `@next/mdx`.**

Rationale: MDX files live under `content/posts/` as a content system, not as route pages inside `app/`. The App Router is the rendering layer; MDX is the data. This matches the `next-mdx-remote/rsc` use case exactly.

**`@next/mdx`** is for when `.mdx` files themselves are route-level pages in the App Router file tree. That approach couples content to routing and makes pagination/filtering harder.

**`next-mdx-remote/rsc`** loads MDX from anywhere (filesystem, CMS, database) and renders it server-side. It works natively with React Server Components in Next.js 15+/16.

**Known limitation:** `next-mdx-remote` has weaker support for dynamic imports and exports within MDX. For Chainless, this is fine — custom components are mapped via the `components` prop, not imported within MDX files.

Install: `npm install next-mdx-remote`

Source: [Blixamo - next-mdx-remote/rsc vs @next/mdx](https://blixamo.com/blog/next-mdx-remote-rsc-vs-next-mdx-nextjs-15), [Next.js MDX Guide](https://nextjs.org/docs/app/guides/mdx)

## MDX Component Mapping

Map custom components to MDX elements via the `components` prop of `MDXRemote`:

```tsx
const components = {
  h2: ({ children, id }) => <h2 id={id} className="...">{children}</h2>,
  h3: ({ children, id }) => <h3 id={id} className="...">{children}</h3>,
  blockquote: PullQuote,
  pre: CodeBlock,
  table: ComparisonTable,
  img: ArticleImage,
  // Custom MDX components
  TldrBox,
  CalloutBox,
  Callout: CalloutBox,
};
```

Heading IDs must be generated at build time (via `rehype-slug`) so the TOC can link to them. Use `remark-gfm` for GitHub-flavored markdown (tables, strikethrough, task lists).

## TOC Generation from MDX Headings

**Approach: Extract headings at build time with a remark plugin, render TOC as a separate component.**

Two-step process:

1. **Extract headings** — use a custom function that parses the raw MDX string with a regex or remark AST visitor to collect H2 and H3 elements with their text and generated IDs
2. **Render TOC** — pass the heading list to a `TableOfContents` client component that renders links and tracks active state

```ts
// Heading extraction (server-side, at build time)
function extractHeadings(rawMdx: string): Heading[] {
  const headingRegex = /^(#{2,3})\s+(.+)$/gm;
  const headings: Heading[] = [];
  let match;
  while ((match = headingRegex.exec(rawMdx)) !== null) {
    const level = match[1].length as 2 | 3;
    const text = match[2].trim();
    const id = text.toLowerCase().replace(/[^\w]+/g, '-');
    headings.push({ level, text, id });
  }
  return headings;
}
```

The heading IDs in the TOC must match the IDs generated by `rehype-slug` in the MDX rendering pipeline. Use the same slugification algorithm in both places.

Sources: [ClarityDev - Interactive TOC](https://claritydev.net/blog/nextjs-blog-remark-interactive-table-of-contents), [CSS-Tricks - TOC with IntersectionObserver](https://css-tricks.com/table-of-contents-with-intersectionobserver/)

## IntersectionObserver for TOC Scroll Tracking

**Pattern: Observe all heading elements, update active TOC item based on which heading is currently in the viewport.**

```tsx
"use client";

function useActiveHeading(headingIds: string[]) {
  const [activeId, setActiveId] = useState<string>("");

  useEffect(() => {
    const observer = new IntersectionObserver(
      (entries) => {
        // Find the first heading that is intersecting
        const visible = entries
          .filter((e) => e.isIntersecting)
          .sort((a, b) => a.boundingClientRect.top - b.boundingClientRect.top);
        if (visible.length > 0) {
          setActiveId(visible[0].target.id);
        }
      },
      {
        rootMargin: "-80px 0px -70% 0px", // navbar offset top, bottom bias
      }
    );

    headingIds.forEach((id) => {
      const el = document.getElementById(id);
      if (el) observer.observe(el);
    });

    return () => observer.disconnect();
  }, [headingIds]);

  return activeId;
}
```

**Key detail:** The `rootMargin` value is critical. `-80px 0px -70% 0px` means:
- Top offset: 80px (navbar height) — don't count headings hidden behind the sticky nav
- Bottom: -70% — only the top 30% of the viewport triggers activation, so the active heading updates slightly before the user reaches it

Source: [Maxime Heckel - Scrollspy Demystified](https://blog.maximeheckel.com/posts/scrollspy-demystified/)

## JSON-LD Structured Data in Next.js 16

**Approach: Render `<script type="application/ld+json">` in the page component (server component).**

Next.js recommends rendering JSON-LD as a `<script>` tag directly in `layout.tsx` or `page.tsx`:

```tsx
// app/blog/[slug]/page.tsx
export default async function ArticlePage({ params }) {
  const article = await getArticle(params.slug);

  const jsonLd = {
    "@context": "https://schema.org",
    "@type": "BlogPosting",
    headline: article.title,
    datePublished: article.date,
    dateModified: article.updatedAt,
    author: { "@type": "Person", name: article.author },
    publisher: {
      "@type": "Organization",
      name: "Chainless",
      logo: { "@type": "ImageObject", url: "https://chainless.com/logo.png" },
    },
    image: article.heroImage,
    description: article.excerpt,
    mainEntityOfPage: {
      "@type": "WebPage",
      "@id": `https://chainless.com/blog/${article.slug}`,
    },
  };

  return (
    <>
      <script
        type="application/ld+json"
        dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
      />
      {/* Article content */}
    </>
  );
}
```

For FAQPage schema, render a second `<script>` tag alongside the Article schema.

Source: [Next.js JSON-LD Guide](https://nextjs.org/docs/app/guides/json-ld)

## Static Generation with generateStaticParams

```tsx
// app/blog/[slug]/page.tsx
export async function generateStaticParams() {
  const posts = await getAllPosts(); // reads MDX files from content/
  return posts.map((post) => ({ slug: post.slug }));
}

// Optional: 404 for unknown slugs
export const dynamicParams = false;
```

All 30 articles are pre-rendered at build time. Zero runtime cost for content pages. `dynamicParams = false` ensures unknown slugs return 404 instead of attempting SSR.

Source: [Next.js generateStaticParams](https://nextjs.org/docs/app/api-reference/functions/generate-static-params)

## Reading Time Calculation

Simple word-count based estimation. Portuguese text reads at ~200-230 WPM (slightly slower than English due to longer words):

```ts
function calculateReadingTime(content: string): number {
  const words = content.trim().split(/\s+/).length;
  const WPM = 220; // Portuguese reading speed
  return Math.ceil(words / WPM);
}
```

Display as "X min de leitura" (PT-BR convention).

## Tailwind v4 Prose/Typography Styling

**In Tailwind v4, `@tailwindcss/typography` is still supported but configured via CSS imports instead of `tailwind.config.js`.**

```css
/* globals.css */
@import "tailwindcss";
@plugin "@tailwindcss/typography";
```

Then use `prose` classes on the article body container. However, the default prose styles assume light theme. For Chainless's dark theme, override via `prose-invert` plus custom CSS variables:

```css
.prose-chainless {
  --tw-prose-body: rgba(255, 248, 240, 0.82);
  --tw-prose-headings: rgba(255, 248, 240, 0.95);
  --tw-prose-links: #FFC73D;
  --tw-prose-bold: rgba(255, 248, 240, 0.95);
  --tw-prose-counters: rgba(255, 248, 240, 0.5);
  --tw-prose-bullets: rgba(255, 248, 240, 0.4);
  --tw-prose-hr: rgba(255, 248, 240, 0.08);
  --tw-prose-quotes: rgba(255, 248, 240, 0.7);
  --tw-prose-quote-borders: #FFC73D;
  --tw-prose-code: rgba(255, 248, 240, 0.9);
  --tw-prose-pre-code: rgba(255, 248, 240, 0.82);
  --tw-prose-pre-bg: rgba(28, 27, 25, 0.6);
  --tw-prose-th-borders: rgba(255, 248, 240, 0.12);
  --tw-prose-td-borders: rgba(255, 248, 240, 0.06);
}
```

**Alternative:** Skip `@tailwindcss/typography` entirely and hand-roll prose styles in CSS. For a brand-heavy dark theme with specific font pairings (Zodiak headings, Switzer body, IBM Plex Mono code), hand-rolled styles may be cleaner than overriding every prose variable. The prose plugin is most valuable when you want quick defaults — Chainless needs intentional control.

**Recommendation:** Start with `@tailwindcss/typography` + `prose-invert` + custom variable overrides. If the overrides become unwieldy, switch to hand-rolled styles.

Source: [Tailwind CSS Discussion #14120 - Prose in v4](https://github.com/tailwindlabs/tailwindcss/discussions/14120), [tw-prose for v4](https://dev.to/gridou/announcing-tw-prose-a-css-only-typography-plugin-for-tailwind-css-v4-o8j)
